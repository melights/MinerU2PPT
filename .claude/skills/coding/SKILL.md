# Coding Skill (Task Blueprint Executor)

## Purpose

Execute tasks from the latest task blueprint generated by `/code-plan`.

Default behavior:
- When invoked by `/code-plan`, execute the full task chain from the plan unless `--tasks` is provided.

## Input Modes

### Mode A: Plan-driven execution
- Input: `--plan <docs/plan/*-plan.md>`
- Optional: `--tasks <comma-separated task IDs>` to execute only selected tasks.

### Mode B: Targeted task execution
- Use `--tasks` to execute selected tasks from the plan.
- If `--tasks` is omitted, execute all executable tasks in dependency order.

## Plan Contract (Latest code-plan format)

Expect task entries with:
- Subject
- Description (including `PRD Section Refs`)
- Blocked By
- Acceptance

## Task Tracking Protocol (Mandatory)

Before coding, the executor must initialize and use task tracking tools:

1. Convert selected plan items into runtime tasks using `TaskCreate`.
   - `subject`: concise imperative task title from plan item
   - `description`: include plan acceptance and `PRD Section Refs`
   - `activeForm`: present-continuous form (required)
2. Preserve dependencies using `TaskUpdate(addBlockedBy=...)` when plan defines blockers.
3. Before starting implementation of a task, set status to `in_progress` via `TaskUpdate`.
4. After implementation + required validation pass for that task, set status to `completed` via `TaskUpdate`.
5. If blocked or partially done, keep `in_progress` and record the blocker in the task description.
6. Use `TaskList`/`TaskGet` during execution to select next unblocked task and verify latest state.

## Execution Rules

1. Read target task(s) and follow `Subject / Description / Blocked By / Acceptance`.
2. Respect `PRD Section Refs` in task descriptions.
3. Execute tasks in dependency order from `Blocked By`.
4. Stay in scope of selected task(s); do not redesign requirements.
5. Implement code and required tests.
6. Run required checks/tests for touched workflow stage(s):
   - implementation+unit
   - syntax/static checks
   - integration tests
   - e2e/smoke (or explicit N/A rationale)
7. Update docs if task requires docs updates.
8. Return file changes + validation evidence and task status updates.

## Output

- Updated code/tests/docs per task scope
- Validation evidence for executed tasks
- Clear completion status for each executed task

## Guardrails

- Do not implement out-of-scope features.
- Do not weaken or remove tests to force pass.
- Do not skip required checks without explicit user approval.
- If task/PRD conflict is found, stop and ask user for clarification.
